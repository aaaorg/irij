# Backend / TypeScript Engineer Review

**Reviewer:** Senior backend / TypeScript engineer
**Datum:** 2026-05-03
**Scope:** `server/src/**`, `shared/src/**`, `tsconfig.base.json`, `server/tsconfig.json`

## TL;DR

Server kód je **nadprůměrně disciplinovaný**: pure utility (pathfinding, walkable, state helpers) jsou skutečně pure a snadno testovatelné, opcodes a payloady jsou single-source-of-truth ve `shared/`, Goja constraints jsou explicitně okomentované a respektované (spread+reassign, `Record` místo `Map`, `number[]` místo `Uint8Array`). Tickrate konstanty z ADR-007 jsou používané všude, kde mají být. Hlavní rizika: **runtime validace user inputu je ad-hoc a duplikovaná** napříč RPC handlery (žádné schema/zod, manuální `typeof` checks v každém handleru), **error semantika** v RPC nesignalizuje failure správně klientovi (vždy 200 + `{ok:false}`), Goja **nested mutation rule není konzistentně dodržen** v `addPresenceToChunk`/A* MinHeap, a **chybí jakékoliv automatizované testy** přesně tam, kde by je čistá utility vrstva trivially přijala (`pathfinding`, `walkable`, `nearestWalkable`).

## Úspěchy

- **Pure utility vrstva oddělená od Nakama runtime.** `pathfinding.ts` a `walkable.ts` neimportují `nkruntime`, jsou snadno mockovatelné. `findPath`/`nearestWalkable`/`maskFromTiledMap` mají jasné contracty v JSDoc-style komentářích včetně edge cases. Tohle je přesně rozhraní, kterým dáš `vitest` jako první test runner bez infra dependencí.
- **Opcodes a payload types jako single-source-of-truth ve `shared/`.** `irij-shared` re-export bez build kroku (`"main": "./src/index.ts"` + `moduleResolution: Bundler`) funguje v IDE i v esbuild bundle pro server. Drift mezi klient/server je strukturálně eliminovaný.
- **Goja constraints documentovány u zdroje.** `state.ts:9-19` a `walkable.ts:11-21` přesně vysvětlují, proč `Record<>` místo `Map<>`, proč `number[]` místo `Uint8Array` a proč spread+reassign — tohle ušetří příštímu engineerovi (i tobě za 3 měsíce) hodiny debugu.
- **Tickrate konstanty správně použité.** `MOVEMENT_SPEED_TPS_BASE`, `TICK_HZ`, `MAX_PATH_LENGTH_TILES`, `NEAREST_WALKABLE_BFS_RADIUS`, `BROADCAST_CHUNK_RADIUS` — všechno z `irij-shared/constants`, žádné magic numbery v `world.ts`/`movement.ts`. Per ADR-007 kontrakt drží.
- **Strict TS konfig.** `noUncheckedIndexedAccess`, `verbatimModuleSyntax`, `isolatedModules`, `noImplicitOverride` — celý monorepo jede na nejpřísnější rozumné settings; `server/tsconfig.json:types: ['nakama-runtime']` izoluje typy mimo client.

## Code smells & risks

- **P1 — Runtime validace inputu je ad-hoc, duplikovaná, bez schema layeru.** `profile.ts:203-238` (`parseRequest` + `validate` + `isInRange`) a `movement.ts:61-75` (`parseMoveRequest`) jsou ručně psané `typeof`/`Number.isFinite` checky se silent fallback na `null`. Nesdílí helpery, nevracejí lokalizovatelné error detaily, nezvládnou hloubku >1. Při Phase 6+ (combat, inventory, trade) tohle exploduje na 10× duplikovanou logiku. Doporučení: tenký zod-like helper (~50 řádků pure TS, viz reference) v `shared/src/validation/` — Goja přežije, esbuild bundluje.
- **P1 — RPC error semantika neoznačuje failures.** `rpc/world.ts:30-39` a všude v `profile.ts` vrací `JSON.stringify({ok:false,...})` na 200 OK. Nakama RPC API podporuje druhý `error` return slot (handler může throwit `Error` s `code` a Nakama vyvolá HTTP/RPC error odpověď), což klient ergonomicky odlišuje od success path. `errorResponse()` helper v `profile.ts:253` zabalí typed error code do success body — klient si pak musí parsovat string + diskriminovat union; throw + Nakama-native error je čistší, log severity grade automatický.
- **P1 — Goja nested mutation rule porušený v MinHeap a chunk bucketu.** `pathfinding.ts:52-112` `class MinHeap` mutuje `this.data` přes index assignment uvnitř single match call — tady je to OK, instance neopouští `findPath`. ALE `state.ts:84` `state.presencesByChunk[key] = bucket;` sice spread+reassignuje bucket, ale potom v `removePresenceFromChunk:97` `delete state.presencesByChunk[key]` je delete nested key na top-level mapě, ne reassign celé mapy. Per komentář na `state.ts:9-19` je to potenciálně lossy. Konzistentní rule: `state.presencesByChunk = { ...state.presencesByChunk }; delete state.presencesByChunk[key];` — i když to vypadá redundantně, drží invariant.
- **P1 — `as` casty bez runtime ověření v `world.ts`.** `world.ts:52` `mapJson as any`, `world.ts:114` `playerObj.value as Player`, `profile.ts:63-69` `playerObj.value as Player`/`skillsObj.value as {...}`. Storage blob z předchozí verze schématu projde jako platný `Player` a server tichounce poběží s missing fields → runtime crash o 10 minut později v jiném handleru. Doporučení: po `storageRead` runtime check klíčových polí (alespoň `id`, `current_position`, `display_name`) → log + reject join, ne tichý cast.
- **P2 — Logging je `printf-style`, ne structured.** `logger.info(\`move rejected userId=${...} reason=${...}\`)` napříč `movement.ts`. Nakama logger podporuje second-arg structured fields (`logger.info('move rejected', {userId, reason})`), které jdou do JSON logu — pro production grep/Loki query je to noc a den. Aktuální format se neparsuje, telemetrický pipeline (Phase 21) bude muset všechno přepsat.
- **P2 — `world.ts:matchLoop` při path-cleanup nereassignuje `presencesByChunk`.** `movement.ts:advanceMovement:308` volá `movePresenceBetweenChunks` který vnitřně dělá `addPresenceToChunk`+`removePresenceFromChunk`. Když path končí (`finished=true`) na hranici chunků, drží se Goja invariant. ALE `state.presencesByUserId = {...}` se reassignuje — `presencesByChunk` mutace je potenciálně visible jen díky tomu, že `addPresenceToChunk` reassignuje bucket. Není to bug dnes, ale fragile contract. Refactor: extract `transactionallyMovePresence(state, userId, newPos)` který reassignuje OBĚ top-level mapy v jednom kroku.

## Doporučené akce

1. **Přidat `vitest` na pure utily** (1-2 h). `server/package.json` `test` script + `server/src/match/*.test.ts` pro `findPath` (no-corner-cutting, octile cost, maxNodes), `nearestWalkable` (BFS depth radius), `maskFromTiledMap` (gid 3 = water = non-walkable). Žádný runtime mock, čistá data in/out. Coverage check není nutný, jde o regression net pro Phase 6+ refactor.
2. **Tenký zod-like validator v `shared/src/validation/`** (~50 řádků, ~3 h). API: `const moveRequestSchema = obj({target: obj({x:int(),y:int()}), client_seq:int()})` → `parse(unknown): Result<T, string[]>`. Nahradí ručně psané `parseMoveRequest` a `parseRequest`/`validate` v profile. Sdílený mezi server (input) a klient (form validace).
3. **Konvertovat RPC error returns na thrown Errors s typed code.** `errorResponse('username_taken')` → `throw new RpcError('username_taken')` + Nakama runtime mapping. Klient pak rozliší success/failure HTTP status, logger.severity v Nakama console je správně warn/error místo info.
4. **Storage blob version + runtime narrowing helper** (1 h). Přidat `schema_version: 1` do Player blobu (už teď, levně) + `function asPlayer(value: unknown): Player | null` který kontroluje klíčová pole. Nahradí `as Player` casty v `matchJoin` a `profileGetSelf`. Migrační cesta pro Phase 5 autosave je triviální.
5. **Refactor `state.ts` na transactionally-safe API**. Místo `addPresenceToChunk` + `removePresenceFromChunk` jako dvou separátních spread-reassignů → single `updatePresenceLocation(state, userId, newPos)` který reassignuje `presencesByChunk` jednou na konci. Eliminuje potenciální Goja race window mezi remove a add.
6. **Structured logger wrapper** (~30 řádků, ~1 h). `function log(logger, level, msg, fields)` → `logger[level](msg + JSON.stringify(fields))` jako bridge dokud Nakama runtime nepodporuje structured `Field` API přímo. Nahradí printf style; v Phase 21 pipeline jen bridge přepneš.

## Reference

- TypeScript 5.x release notes (verbatim module syntax, isolatedModules) — relevantní pro shared package re-export bez build (žádné typy přes `export *` mimo `export type`, což shared dnes dodržuje, viz `messages/index.ts`).
- Nakama TS runtime examples (heroiclabs/nakama-project-template): `src/main.ts` jako single InitModule, RPC throw-Error pattern pro typed errors, `nk.storageRead()` value-as-unknown narrowing.
- Goja Export()/Import() semantics — `runtime_javascript_match_core.go` `stateObject.Set(k, v)` per-callback rebuild; vysvětluje, proč `Map`/`Set`/typed arrays neprojdou (state.ts:9-19 to už dokumentuje).
- Zod-lite reference: pattern `Result<T, E>` validátorů jako čistý TS bez závislostí (~50 LOC) — kompatibilní s Goja, žádný `import.meta`, žádný regex /v flag.
