# i18n / Lokalizační specialista

## TL;DR

i18next + browser-languagedetector jsou v `client/package.json`, ale **nikde** ve `client/src` nejsou wired up — žádný `i18n.init()`, žádné `t()` calls, žádný `locales/` adresář. Všechny user-facing stringy v `LoginScene` a `CharacterCreationScene` jsou hardcoded česky (`'Hrát jako host'`, `'Sjednávám spojení s Iriji…'`, dictionary `ERROR_MESSAGES`). Phase 17 plánuje setup až po Phases 5-16, což znamená rostoucí refactor dluh: každá nová scéna (Phases 5/6/7/12-16) přidá hardcoded stringy. Server posílá `code` enum (`invalid_username`, `username_taken`) — to je **správný pattern** pro i18n, ale klient ho dnes překládá inline v scéně místo přes `t('error.invalid_username')`. Také je drift mezi ADR-016 (`{ cs, en }` per dialog node, EN od MVP) a `02d` (`text_cs`/`text_en`, EN „odsunuto post-MVP"). Doporučuji **posunout i18n setup z Phase 17 na Phase 5a** (před tím, než Phase 6+ přidají další UI).

## Co je dnes ready

- ADR-016 je jasný: CS+EN od MVP, i18next, ICU MessageFormat, CI lint na key parity, fallback chain `navigator.language` → CS, persist v `Player.settings`. Plán je dobře promyšlený.
- Server-side errory už jsou strukturované jako string enums (`CreateCharacterError = 'invalid_username' | ...` v `shared/src/messages/profile.ts`, `MoveRejectReason` v `movement.ts`). To je textbook pattern pro i18n — stačí mapovat `code` → `t('error.' + code)` na klientu.
- Phaser používá `Scene.add.text` (Web Canvas, system fonts), ne `BitmapText` — UTF-8 háčky/čárky jdou přímo bez glyph map. Když přijde EN pass, není fontové riziko.
- Dependencies správně zvolené (`i18next ^26`, `i18next-browser-languagedetector ^8.2`); kompatibilní s Vite ESM build.

## Rizika

- **P0 — Drift ADR-016 vs `02d-data-model-npc-mobi-questy.md`.** `02d` (řádek 110) říká `text_cs`/`text_en` flat keys a „MVP jen `cs`". ADR-016 (řádek 494) říká `{ cs, en }` nested struktura a EN od MVP. Inkonzistence způsobí, že první implementer si vybere a druhý bude refaktorovat. Sjednoť jeden tvar (preferuju `{ cs, en }` nested = lepší pro ICU + víc jazyků).
- **P0 — i18n posunutý do Phase 17 = rostoucí dluh.** Dnes 2 scény mají ~30 hardcoded CS stringů. Po Phase 5-16 (persistence, combat, inventory, gathering, crafting, NPC, quests, market, chat) odhadem 300-500 stringů. Refactor v Phase 17 = ~1 týden, ne 2-3 dny. **Setup i18n teď** (cs.json + `t()` wrapper), EN překlad odsuň do Phase 17 — ale klíčová struktura ať existuje od dne 1.
- **P1 — Plurálky CS jsou 1/2-4/5+/decimal**, en má 1/other. Bez ICU MessageFormat (`{count, plural, one {…} few {…} other {…}}`) brzy uvidíš `"5 minci"` nebo `"1 minci"`. i18next sám má základní plural support; pro CS doporučuju `i18next-icu` plugin + `intl-messageformat`.
- **P1 — Server-side notifikace.** Match handler dnes broadcastuje raw `code` v `MOVE_REJECTED.reason`. Phase 17 task zmíní „thin i18n wrapper pro server-side notifikace" — ale **server vůbec nemá vědět o jazyce hráče**. Posílej jen kódy + parametry (`{ code: 'inv_full', params: { capacity: 24 } }`), klient si zformátuje. To `Player.settings.locale` server jen ukládá pro mailové notifikace post-MVP.
- **P2 — Vokativ a slovanský charakter.** „Ahoj, {name}" ↔ „Ahoj, Jakube" — vokativ je v CS gramaticky povinný v dialozích. Buď omez NPC dialog na nominativ (`„Ahoj, cestovateli"`), nebo přidej vokativ field do `Player` (ne dnes, ale pamatuj). PL/SK jsou podobně skloňované; EN to nemá.

## Doporučené akce

1. **Phase 5a** (před persistence): proveď základní i18next setup — `client/src/i18n.ts`, `cs.json` s ~30 existujícími stringy, refactor `LoginScene` + `CharacterCreationScene` na `t()`. ~1 den. EN překlad odložený.
2. **Sjednoť dialog format** v `02d` na `{ cs: "...", en: "..." }` per node = match ADR-016. Update PR + changelog v 02d.
3. **Definuj error namespace strategy** v `shared/src/messages/`: každý error code má kanonický key `error.<domain>.<code>` (např. `error.profile.invalid_username`). Klient mapuje 1:1, server posílá pure code. Žádné texty na drátě.
4. **Přidej `i18next-icu` + plugin** od dne 1, i kdyby všechny stringy byly nominativní — když ho přidáš později, refactor plurálových stringů je tichý a snadno se zapomene.
5. **CI lint script** (Phase 17): `pnpm i18n:check` jako Node skript co diffuje `cs.json` ∩ `en.json` keys. Žádný heavy framework. Plus `eslint-plugin-i18n-json` pro JSON validitu.
